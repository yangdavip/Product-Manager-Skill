# 需求分析模块

> 产品经理核心能力之一：需求分析与业务目标定义
> 参考标准：CMMI 3.0 REQM（需求管理）

---

## 适用场景

- 新项目/新功能的需求调研
- 用户访谈、竞品分析
- 需求评估与优先级排序
- 业务目标定义
- 产品规划与路线图制定

---

## 需求分析全流程

```
需求收集 → 需求分析 → 需求定义 → 需求验证 → 需求管理
   ↓           ↓           ↓           ↓           ↓
用户调研     用户画像     业务目标     需求评审     需求追踪
竞品分析     需求拆解     角色权限     验收标准     变更管理
数据分析     场景分析     状态流转     原型验证     版本管理
```

## 一、需求收集方法

### 1.1 用户访谈

**访谈准备**
| 准备项 | 说明 |
|--------|------|
| 访谈目标 | 明确要了解的问题 |
| 访谈对象 | 选择典型用户（新手/熟练/专家） |
| 访谈提纲 | 准备结构化问题清单 |
| 记录方式 | 录音/笔记/观察 |

**访谈问题模板**
```
1. 背景问题
   - 您的日常工作流程是什么？
   - 您在工作中遇到的最大挑战是什么？

2. 现状问题
   - 您目前如何处理XXX？
   - 这个过程中有哪些痛点？

3. 期望问题
   - 您希望系统如何帮助您？
   - 如果有一个理想工具，它应该具备什么功能？

4. 优先级问题
   - 在这些需求中，哪个对您最重要？为什么？
```

### 1.2 问卷调研

**问卷设计原则**
- 问题简洁明了
- 避免引导性问题
- 混合开放/封闭式问题
- 控制问卷长度（10-15分钟）

**问卷结构**
```
1. 基本信息（2-3题）
2. 现状评估（3-5题）
3. 需求优先级（3-5题）
4. 开放建议（1-2题）
```

### 1.3 竞品分析

**竞品分析框架**
| 分析维度 | 分析内容 |
|----------|----------|
| 功能对比 | 核心功能、差异化功能 |
| 用户体验 | 交互设计、操作流程 |
| 商业模式 | 定价策略、目标市场 |
| 技术实现 | 技术栈、性能指标 |
| 优劣势 | SWOT分析 |

**竞品分析模板**
```markdown
| 功能点 | 竞品A | 竞品B | 竞品C | 我们 |
|--------|-------|-------|-------|------|
| 功能1 | ✅ | ✅ | ❌ | ✅ |
| 功能2 | ✅ | ❌ | ✅ | ❌ |
| 用户评分 | 4.5 | 4.0 | 3.5 | - |
```

### 1.4 数据分析

**数据分析维度**
| 数据类型 | 分析指标 | 分析目的 |
|----------|----------|----------|
| 用户行为 | PV/UV、停留时间、跳出率 | 了解用户使用习惯 |
| 业务数据 | 转化率、留存率、流失率 | 评估业务健康度 |
| 性能数据 | 响应时间、错误率、可用性 | 评估系统性能 |
| 反馈数据 | 工单、投诉、建议 | 发现用户痛点 |

## 二、用户分析

### 2.1 用户画像（Persona）

**用户画像模板**
```markdown
## 用户画像：[角色名称]

### 基本信息
- **姓名**：张三（虚构）
- **年龄**：30岁
- **职位**：运营经理
- **经验**：5年行业经验

### 工作场景
- **日常工作**：负责园区企业入驻、合同管理、费用结算
- **工作目标**：提高运营效率，降低管理成本
- **工作痛点**：手工操作多、数据分散、易出错

### 使用习惯
- **使用频率**：每天使用
- **使用场景**：办公时间，多任务切换
- **技术水平**：中等，熟悉常用办公软件

### 需求期望
- **核心需求**：自动化处理、数据可视化
- **期望效果**：减少手工操作50%，提高效率
- **使用障碍**：学习成本高、操作复杂
```

### 2.2 用户角色矩阵

**角色定义模板**
```markdown
| 角色 | 说明 | 典型用户 | 使用频率 | 核心目标 |
|------|------|----------|----------|----------|
| 系统管理员 | 系统配置、用户管理 | IT人员 | 低频 | 系统稳定运行 |
| 运营管理员 | 日常业务操作 | 运营人员 | 高频 | 提高工作效率 |
| 财务人员 | 财务相关操作 | 财务人员 | 中频 | 准确核算费用 |
| 普通用户 | 基础功能使用 | 企业用户 | 中频 | 便捷办理业务 |
| 管理层 | 数据查看、决策 | 部门经理 | 低频 | 掌握经营状况 |
```

### 2.3 用户旅程图（User Journey Map）

**用户旅程图模板**
```markdown
## 用户旅程：[场景名称]

### 阶段1：认知阶段
- **用户行为**：了解产品/功能
- **用户想法**：这个产品能帮我解决什么问题？
- **用户情绪**：😐 中立
- **痛点**：信息不清晰
- **机会**：提供清晰的产品说明

### 阶段2：使用阶段
- **用户行为**：开始使用功能
- **用户想法**：如何操作？在哪里找到？
- **用户情绪**：😕 困惑
- **痛点**：操作复杂
- **机会**：简化操作流程

### 阶段3：完成阶段
- **用户行为**：完成任务
- **用户想法**：完成了吗？结果正确吗？
- **用户情绪**：😊 满意
- **痛点**：反馈不及时
- **机会**：提供即时反馈

### 阶段4：复购阶段
- **用户行为**：再次使用
- **用户想法**：下次还用这个产品
- **用户情绪**：😊 信任
- **痛点**：无
- **机会**：持续优化体验
```

## 三、需求分析方法

### 3.1 需求分类

**需求分类框架**
```markdown
需求
├── 业务需求（Business Requirements）
│   ├── 业务目标
│   ├── 业务规则
│   └── 业务约束
├── 用户需求（User Requirements）
│   ├── 功能需求
│   ├── 体验需求
│   └── 场景需求
├── 系统需求（System Requirements）
│   ├── 功能需求
│   ├── 非功能需求
│   │   ├── 性能需求
│   │   ├── 安全需求
│   │   ├── 可靠性需求
│   │   └── 兼容性需求
│   └── 接口需求
└── 约束条件（Constraints）
    ├── 技术约束
    ├── 业务约束
    ├── 法规约束
    └── 资源约束
```

### 3.2 需求拆解

**需求拆解模板**
```markdown
## 需求拆解：[需求名称]

### 业务需求
- **业务目标**：提高XX效率30%
- **业务规则**：XX规则、XX规则
- **业务约束**：必须在XX时间内完成

### 用户需求
- **功能需求**：
  - FR-001：查看列表
  - FR-002：新增数据
  - FR-003：编辑数据
- **体验需求**：
  - 操作简单，3步内完成
  - 响应时间<2秒
- **场景需求**：
  - 场景1：日常操作
  - 场景2：批量处理

### 系统需求
- **功能需求**：见SRS文档
- **非功能需求**：
  - 性能：支持1000并发
  - 安全：RBAC权限控制
  - 可用性：99.9%
- **接口需求**：
  - 对接XX系统
  - 对接XX接口

### 约束条件
- **技术约束**：使用XX技术栈
- **业务约束**：必须兼容现有流程
- **法规约束**：符合XX法规
- **资源约束**：开发周期3个月
```

### 3.3 需求场景分析

**场景分析模板**

```markdown
### 场景描述
```

## 四、业务目标定义

### 4.1 问题描述

```markdown
##### 问题描述
- **现状**：当前存在什么问题？（效率低/易出错/成本高/体验差）
- **影响范围**：影响哪些用户/部门/业务？
- **量化数据**：当前问题的量化指标是什么？（如：每月处理1000笔，错误率5%）
- **根因分析**：问题的根本原因是什么？
```

### 4.2 解决方案

```markdown
##### 解决方案
- **核心思路**：如何解决这个问题？
- **关键功能**：需要哪些核心功能？
- **技术方案**：采用什么技术实现？
- **预期效果**：上线后期望达到什么效果？
```

### 4.3 成功标准

```markdown
##### 成功标准
| 指标 | 当前值 | 目标值 | 衡量方式 | 负责人 |
|------|--------|--------|----------|--------|
| 效率提升 | 处理时间30分钟 | 处理时间5分钟 | 系统日志 | |
| 错误率降低 | 5% | 0.5% | 数据统计 | |
| 用户满意度 | 60分 | 90分 | 问卷调查 | |
| 成本节约 | 10万/月 | 2万/月 | 财务数据 | |
```

### 4.4 业务价值分析

```markdown
##### 业务价值分析
| 价值维度 | 预期收益 | 计算方式 |
|----------|----------|----------|
| 效率提升 | 节约XX人天/月 | 人均工时×减少工时 |
| 成本降低 | 节约XX万元/月 | 减少人力+减少错误 |
| 收入增长 | 增加XX万元/月 | 提高转化率×客单价 |
| 用户体验 | 满意度提升XX% | NPS评分提升 |
```

## 五、优先级评估

### 5.1 MoSCoW方法

| 优先级 | 说明 | 占比 | 示例 |
|--------|------|------|------|
| **Must Have** | 必须有，核心功能 | 40-60% | 登录、核心业务流程 |
| **Should Have** | 应该有，重要功能 | 20-30% | 数据导出、报表统计 |
| **Could Have** | 可以有，锦上添花 | 10-20% | 个性化设置、主题切换 |
| **Won't Have** | 不做，超出范围 | 0% | 远期规划功能 |

### 5.2 RICE评分法

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| **Reach**（覆盖面） | 25% | 影响用户数量 |
| **Impact**（影响力） | 30% | 对用户的影响程度 |
| **Confidence**（信心度） | 20% | 对估算的信心程度 |
| **Effort**（工作量） | 25% | 实现所需工作量 |

**RICE评分公式**
```
RICE = (Reach × Impact × Confidence) / Effort
```

**评分示例**
```markdown
| 需求 | Reach | Impact | Confidence | Effort | RICE | 优先级 |
|------|-------|--------|------------|--------|------|--------|
| FR-001 | 1000 | 3 | 90% | 5 | 540 | 高 |
| FR-002 | 500 | 2 | 80% | 3 | 267 | 中 |
| FR-003 | 200 | 1 | 70% | 2 | 70 | 低 |
```

### 5.3 优先级评估矩阵

```markdown
| 需求ID | 需求名称 | 业务价值 | 实现成本 | 风险程度 | 紧急程度 | 综合评分 | 优先级 |
|--------|----------|----------|----------|----------|----------|----------|--------|
| FR-001 | | 高 | 中 | 低 | 高 | 85 | P0 |
| FR-002 | | 中 | 低 | 低 | 中 | 65 | P1 |
| FR-003 | | 低 | 高 | 高 | 低 | 35 | P2 |
```

## 六、需求验证

### 6.1 需求评审检查

**评审检查清单**
```markdown
| 检查项 | 检查内容 | 是否通过 |
|--------|----------|:--------:|
| 完整性 | 所有需求是否都已收集？ | □ |
| 一致性 | 需求之间是否矛盾？ | □ |
| 可行性 | 技术上是否可实现？ | □ |
| 可测试性 | 是否有明确的验收标准？ | □ |
| 必要性 | 是否都是必需的？ | □ |
| 优先级 | 优先级是否合理？ | □ |
```

### 6.2 原型验证

**原型验证流程**
```
低保真原型 → 用户测试 → 收集反馈 → 优化设计 → 高保真原型 → 再次验证
```

**用户测试模板**
